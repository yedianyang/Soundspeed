## 1. 实际链路

```
POST /notes (202 async)
  → parse_note()  只抽 @category，丢弃 scene/take 前缀（takes.py:297 注释自述）
  → orchestrator.run_np_async()  fire-and-forget
      → _run_np_async()  查 DB 拼 take_context 注入 NPInput
          → run_np_note()  LLM 推理 → {take_id, category, content}
          → dal.insert_note()  写 manual.note + 重建 takes.notes 聚合
          → WS publish note.processed
      → _np_done_callback()  失败时仅 log WARNING（无降级写）
```

涉及文件：

- `backend/pipelines/np_note.py` — NPInput/NPOutput/run_np_note，LLM prompt builder + JSON 解析
- `backend/core/orchestrator.py:426-575` — run_np_async / _run_np_async / _np_done_callback / create_orchestrator
- `backend/api/routes/takes.py:284-335` — POST /notes
- `backend/db/dal.py:650-707` — insert_note / list_notes
- 前端 `NoteMemo.tsx` / `NoteList.tsx` / `useLiveConnection.ts` / `store/session.ts`

## 3. 主问题

一句话概括：**该确定的地方交给了不确定，不确定失败了又不告诉人。**

`note_parse` 已经能确定性解析出 `3A 2` 这种前缀，但 API 把它丢了（takes.py:297「只提取 @category，不定位 take」），take 定位一律交给 LLM 猜。而 LLM 任何失败都静默丢 note。

### 3.1 失败静默丢失（代码可见事实，UX 黑洞）

- 后端：`_np_done_callback`（orchestrator.py:531-550）的 docstring 自述「失败：记 WARNING + 发 idle。note 丢失（无降级写），日志留痕」。LLM 报错时不写库、不通知前端。
- 前端：移除 pending note 的唯一路径是收到 `note.processed`（session.ts:203-208，按 `ts+category+content` 三元组匹配）。失败时后端只发 `llm.status idle`（useLiveConnection.ts:47-50，只 setLlm，不碰 pendingNotes）。**没有 note.failed 事件，也没有 setTimeout 超时清理。**
- 合并效果：LLM 一失败，那条 note 永远挂在灰色「处理中」态，永不落地、永不报错，刷新页面后凭空消失。而用户已经收到 202，以为成功了。

兜底逻辑其实写了，但只在「不在 event loop」的测试分支用（takes.py:310-329 fallback 到当前活跃 take），LLM 失败的生产路径没复用它。

### 3.2 确定性前缀被丢弃 + 409 契约退化（事实 + 静态推演）

- 事实：`parse_note` 解析出的 scene_code/take_number 在 API 层未被使用，take 定位完全交给 LLM。明确的 `3A 2` 前缀本该是最可靠的信号，却交给 LLM 推断。
- 静态推演（未实跑）：无活跃 take 且无历史时，`take_context` 为空、`current_take_id=None`，LLM 被 system prompt 强制输出 `take_id:int`，只能瞎编。`take_events.take_id` 有外键 `REFERENCES takes ON DELETE CASCADE`（schema.sql:63-64，`foreign_keys=ON`），会挡下 `IntegrityError` → 落进 3.1 的静默丢失。
- 后果：外键保证不写脏数据（好），但 spec §2.2 明定的「无活跃 take → 409」退化成了「202 + 后台默默失败」。契约从「同步报错」变成「假成功」。

### 3.3 NL→take 实体解析层不可复用（违背反复声明的架构要求）

- 4.A spec 与 4.B 任务卡三次强调「NL→实体解析层做成可被 QP（7.x）复用」。
- 现状：指代消解（「这条」「上一条」「3A 2」）埋进了 NP 的 LLM system prompt（np_note.py:48-64）。QP（7.x）无法复用这套逻辑。rule-based 的 `note_parse` 只解析结构化前缀，且在 API 被丢弃。
- 这点不在 flowchart 里，按用户的锚算次级，但它有下游影响（7.x QP 会重复造轮子），单独拎出。

## 4. 文档债（次级，要动的是 spec 不是代码）

- 本地 spec `docs/specs/2026-06-12-note-input-design.md` 标着「草稿，待 Lead 评审」，从未批准。里面写的是 rule-based parser + 201 同步响应。实现回归了 flowchart 的 LLM + 202 async，绕过了这份未批草稿。**结论是 spec 落后于代码，应升 v0.2 记录 LLM 路线 + 202 契约**，而非代码超纲。
- 副作用：`note_parse` 的 scene/take 正则解析现在是死代码（解析了不用）。
- event_type：实现用 `manual.note`（dal.py:674），flowchart 与 4.B 任务卡写的是 `system.note`，spec v0.1 已改成 `manual.note` 且理由充分（system.note 留给系统自动事件）。统一到 `manual.note` 即可，但要同步 flowchart/任务卡措辞。

### 测试盲区

NP 失败链路无集成测试覆盖。`test_np_note.py` 只测纯函数层（`_parse_llm_output` 的 JSON 解析 / category 校验 / take_id 类型，`_build_user_message` 的 prompt 拼装）。以下路径全无测试：take_id 幻觉落库失败、无活跃 take 兜底、失败降级 / 前端通知、`_run_np_async` 全链路集成。「288 绿」不代表失败路径被验证，是它压根没测。

## 6. 建议处置（按优先级）

1. **失败可见**（对应 3.1）：加 `note.failed` WS 事件或让前端 pending note 超时翻红；LLM 失败时复用 takes.py:310 的 rule-based 兜底（写当前活跃 take），而非静默丢。
2. **确定性优先**（对应 3.2/3.3）：明确的 `3A 2` 前缀走 rule-based 直接定位 take_id，只有模糊指代（「这条」「上一条」）才交 LLM。顺带让实体解析层可被 QP 复用。
3. **补失败链路集成测试**（对应测试盲区）：红用例先行，覆盖 LLM 失败 / take_id 不存在 / 无活跃 take。
4. **spec 升 v0.2**（对应 §4）：记录 LLM 路线 + 202 契约，清理 note_parse 死代码或明确其定位。
