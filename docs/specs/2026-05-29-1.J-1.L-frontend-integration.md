## 1. 背景：1.I 留下的两个缺口

1.I 交付了 FastAPI app 工厂 + take.start/end + WS 转发，但：

1. **没有 GET 端点**：`GET /api/v1/takes`、`GET /api/v1/takes/{id}` 被 1.I 显式跳过。1.L 的初始列表加载 / 详情依赖它们。
2. **没有可运行入口**：仓库里没有 uvicorn / `__main__`，`create_app(orchestrator)` 没有任何地方调用 → 后端起不来，无法做各 ticket 验收要求的「浏览器手动验证」。

DAL 读方法齐全，补齐成本小。Lead 决策（2026-05-29）：在本分支一起补，附本 delta spec 留痕。

### 1.1 上游依赖现状（codex P0）

`asr.partial/final.ch{1,2}` 目前**没有生产者**——1.A（ASRService 双路转录）/ 1.C（结构化输出 + publish）未完成，入口链路里没有任何组件 publish asr 事件。影响：

- 1.J transcript 面板按合约接 WS asr.* topic 是**完整可做**的（topic + payload 形状已由 1.I 定死），但**真实音频→转录的 e2e 数据源要等 1.C**。
- 为让 1.J 在 1.C 之前可视化验收，本分支加一个 **dev-only 合成 ASR 注入端点**（见 §2.4），手动驱动 transcript 面板。
- L2 在空 segments 上跑会得到空摘要——属预期降级，不阻塞 1.K/1.L 链路验证。

## 2. 后端补齐项（cross-lane，backend/api + backend/core + backend/llm）

### 2.1 GET 端点（`backend/api/routes/takes.py`，async def）

鉴权沿用 `require_admin`。**必须 async def**（codex P2）：DAL 单连接 `check_same_thread=False`，同步路由跑线程池会与 event-loop 线程上的 L2 async 任务并发操作同一 `_conn`（SQLite 对同一连接对象的并发调用不安全），理由同 takes.py 决策 1。

- `GET /api/v1/takes?scene_id=<int|省略>` → `{takes: [TakeDTO, ...]}`
  - 省略 scene_id 返回全部（`dal.list_takes(None)`）。不含 segments（避免 N+1）。
- `GET /api/v1/takes/{take_id}` → `TakeDetailDTO`，take 不存在 → 404。

**TakeDTO**（从 `dal.Take` 投影，snake_case，前端 boundary 映射）：

```
take_id:int, scene_id:int, take_number:int, shot:str|null,
start_ts:float, end_ts:float|null, status:str('keeper'|'ng'|'hold'|'tbd'),
script_diff:object|null, notes:str|null, created_at:float, updated_at:float
```

> **有意省略**（codex P2）：`performer_issues`、`audio_quality` 属 NP Pipeline 输出，1.J–1.L 不暴露，故意不进 DTO。

**TakeDetailDTO** = TakeDTO + `segments:[{segment_id, ch, speaker, text, start_frame, end_frame}]`。

> **不再单列 line_matches**（codex P1）：`script_diff`（L2 写入）已自带 `line_matches`（含 `line_no`、含 insertion）。`dal.list_take_line_matches` 返回的是另一种形状（`line_id`、无 `line_no`、过滤了 insertion），双来源同名歧义。前端一律从 `script_diff.line_matches` 取。

> 注：take 的 `status` 由用户标记决定，本阶段无「改 status」端点。L2 只产出 `script_diff`，**不改 status**——接入后 take.status 停在 `tbd`，前端 mark/状态 badge 暂留本地 override。

### 2.2 llm.status 事件（驱动前端 LLM 黄点 Loading）

用户要求：Gemma **首次使用时才加载**（LLMService 已 lazy init），加载/运行期间前端 LLM chip 显示黄点。

- `backend/core/events.py`：`LLM_STATUS = "llm.status"` + `LlmStatusPayload(state:str, task_type:str|None, take_id:int|None)`，`state ∈ {"idle","loading","running"}`。
- `backend/llm/service.py`：加只读属性 `model_loaded:bool`（`self._client is not None`），供 orchestrator 区分首次 loading vs 后续 running。
- `backend/core/orchestrator.py`：
  - `_run_l2_async` **起手**publish `llm.status`：`getattr(llm_service, "model_loaded", True)` 为假 → `loading`，否则 `running`。
  - **idle 统一在 `_l2_done_callback` 发**（codex P2）：callback 无论成功/失败都 publish 一次 `idle`；`_run_l2_async` 内不发 idle。消除双发歧义。
- `backend/api/app.py`：lifespan 把 `LLM_STATUS` 也订阅到 `cm.broadcast`。

前端：`idle` → 绿点 Idle；`loading` → 黄点 Loading…；`running` → 黄点 L2。

> **已知风险（codex P2，不在本分支修）**：`GemmaClient.__init__` 在 `_worker` 协程内同步调用（`_ensure_client` 不在 `to_thread`），首次加载权重会冻 event loop 数秒（期间 WS/REST 假死）。`loading` 在阻塞前已 publish，UX 意图成立但体验有顿挫。彻底修需把 `_ensure_client` 挪进 `to_thread`（1.F lane），本分支只记录，交后端 owner 决定。

### 2.3 可运行入口（`backend/api/__main__.py`）

`python -m backend.api` 启动：

- 读 env：`SOUNDSPEED_DB`（默认 `./soundspeed.db`）、`ADMIN_TOKEN`（缺失走 1.I 随机生成 + console 打印）、`HOST`（默认 `0.0.0.0`）、`PORT`（默认 `8000`）、`SOUNDSPEED_DEV`（=1 时挂 §2.4 dev 注入端点）。
- `dal = DAL(db_path)` → `llm_service = get_service()`（**单例**，codex P1，不裸 `LLMService()`，lazy 不加载模型）→ `orchestrator = create_orchestrator(dal, llm_service=llm_service)`（自动绑 `run_l2_take`）→ `app = create_app(orchestrator, llm_service=llm_service)` → `uvicorn.run(app, host, port)`。
- **lifespan 关闭时 `await llm_service.aclose()`**（codex P1）：`create_app` 新增可选 `llm_service` 参数存 `app.state`，lifespan `yield` 后调 `aclose()`，避免 `_worker_task` 泄漏。

### 2.4 dev-only 合成 ASR 注入端点（codex P0，仅 SOUNDSPEED_DEV=1 挂载）

`POST /api/v1/debug/asr`（require_admin），body `{ch:int, text:str, speaker:str|null, is_partial:bool}` → 服务端组 `AsrPartialPayload`/`AsrFinalPayload`（start/end_frame 服务端补）→ `orchestrator.publish(asr.{partial|final}.ch{ch})`。仅用于 1.C 落地前手动驱动 1.J transcript 面板验收。生产（SOUNDSPEED_DEV 未设）不挂载。

### 2.5 scene 来源（v0.3，smoke 发现）

`takes.scene_id` 是 `NOT NULL REFERENCES scenes(scene_id)` 且 `PRAGMA foreign_keys=ON`，但全仓没有 create-scene 端点。1.K 起 take 必须有一个已存在的 scene_id，否则 `start_take` 抛 IntegrityError 被 `publish()` 吞掉，take.changed 永不发。补：

- `GET /api/v1/scenes`（async def, require_admin）→ `{scenes: [dal.list_scenes() dict]}`（含 `scene_id/scene_code/description/shoot_date/is_active/created_at`）。
- **DEV 自动播种**（`entrypoint.build_app()` 内，`SOUNDSPEED_DEV=1` 且 `list_scenes()` 为空时）：`create_scene("Scene_1") + set_active_scene`。重启持久 DB 幂等跳过。**只在 build_app（真服务）做，不在 create_app**——避免污染既有 create_app(orch) 测试。
- 前端 1.K：load 时 `GET /scenes`，用 active（或首个）scene 的 id 起 take，并把 scene_code/编号显示为「当前场次」。scene 切换/新建下拉仍本地（scope guard，无 CRUD 端点）。

> debug ASR 注入段（§2.4）的 `start_frame/end_frame`：受 `transcript_segments` 的 `CHECK(end_frame > start_frame)` 约束，取 `start_frame=int(time.time()*1000)`、`end_frame=start_frame+1000`（活跃 take 时正确持久化，供 1.L 详情显示）。

### 2.6 模型缺失自动下载 + llm.status "downloading"（用户请求）

后端无 Gemma 模型时自动下载，前端 LLM chip 显示「Downloading…」。

- `backend/llm/service.py` `resolve_model_path(download)`：优先级 (1) `GEMMA_MODEL_PATH` 设置且文件存在 → 直接返回，**不碰 HF**（用户当前 run 走这条，零网络、快）；(2) `huggingface_hub.try_to_load_from_cache(unsloth/gemma-4-E4B-it-GGUF, gemma-4-E4B-it-Q4_K_M.gguf)` 命中 → 缓存路径；(3) `download=True` → `hf_hub_download`（缓存未命中才真下，带 resume）；(4) 否则 None。
- `LLMService`：`model_present`（同步 cheap：resolve(False) 非 None 或 client 已加载）；`async ensure_model_ready()` = `to_thread(resolve, True)`；`_worker` 里 `client = await to_thread(_ensure_client)` —— **顺带修了首次加载冻 loop 的已知风险（advisor §2.2）**，downloading/loading WS 帧现在能即时送达。
- `orchestrator._run_l2_async`：`if not svc.model_present: publish(downloading) → await ensure_model_ready()`，再走原 loading/running。
- 前端：`LlmStatusMsg.state` 加 `"downloading"`，LLM chip → 黄点「Downloading…」。
- **依赖**：新增运行时依赖 `huggingface_hub`（已装进 `.venv`）。**仓库无依赖清单**（`pyproject.toml` 无 `[project].dependencies`，无 requirements.txt；`fastapi`/`uvicorn`/`llama-cpp-python` 同样只在 .venv）——huggingface_hub 现状一致但未声明，建议团队补一份依赖清单（独立 hygiene ticket）。`resolve_model_path` 对 huggingface_hub 做 lazy import，`GEMMA_MODEL_PATH` 命中时根本不 import，故已有部署（带 GEMMA_MODEL_PATH）不受影响。
- 范围：5GB 真下载未实测（缓存命中即返回，逻辑 + 缓存解析已验证）；进度 % 未做（v1 只「Downloading…」态），可作后续增强。

## 3. 前端接入契约

### 3.1 连接 / 配置

- `frontend/src/lib/config.ts`：`API_BASE`（`import.meta.env.VITE_API_BASE` ?? `http://localhost:8000`），`WS_BASE` 由 API_BASE 派生（http→ws / https→wss）。
- **admin token**：SettingsDialog 新增「服务器连接」段（API base 显示 + token 输入），存 `localStorage`（`soundspeed.adminToken` / `soundspeed.apiBase`）。无 token → 连接置「未鉴权」态，提示去设置填。
- `frontend/.env.example` 记 `VITE_API_BASE`。

### 3.2 REST（`frontend/src/lib/api.ts`，Bearer header）

`getTakes(sceneId?)` / `getTake(id)` / `startTake(scene_id, shot?)` / `endTake()`。react-query 管 getTakes/getTake。

### 3.3 WS（`frontend/src/lib/ws.ts` + `useLiveConnection`）

- 单连接 `${WS_BASE}/ws?token=<token>`，连上收全部 topic（1.I 无客户端订阅协议）。
- 信封：`{topic:string, payload:object}`。topic：`asr.partial.ch{1,2}` / `asr.final.ch{1,2}` / `take.changed` / `llm.status`。
- payload 形状见 `backend/core/events.py`。
- 自动重连（指数退避，封顶）。**重连成功后 invalidate/refetch `getTakes`**（codex P1）：断线期间错过的 take.changed（尤其 L2 那条）靠重取对齐，避免 take 永久停在 `script_diff=null`。

### 3.4 状态（`frontend/src/store/session.ts`，zustand）

- `connection`：`'connecting'|'open'|'closed'|'no-token'`。
- `segments`：当前录制 take 的实时转录（按 ch 维护；partial 替换该声道最后一条 partial，final 落定）。
- `currentTake`：`{take_id, scene_id, take_number, shot, recording}`——`recording` 本地（前端按 start/end），`take_number/take_id` 从 take.start 触发的 take.changed 取。
- `takes`：`Map<take_id, TakeDTO>`，**patch-merge by take_id**（codex P1）：take.changed 只携带 `{take_id, scene_id, take_number, status, script_diff}` 5 字段，**合并覆盖这几个字段到已有条目**，不整对象替换（否则 `shot/start_ts/end_ts/notes` 等会被抹成 undefined）。take_id 不在 map（新 take）→ 先插部分条目，其余字段等 getTakes/getTake 补齐。
- `llm`：`{state}`，来自 llm.status。

### 3.5 各面板替换点

| ticket | 组件 | mock → 真 |
|---|---|---|
| 1.J | LiveTranscript / TakeBlock | mock → store.segments（partial 灰/final 黑，speaker 分色） |
| 1.K | BottomControlBar / AdminHome / StatusChip(LLM) | REC → start/endTake；状态栏 scene/take/录制态；LLM chip ← llm.status |
| 1.L | HistoryTakes / LLMFeedback | mock → react-query getTakes + store.takes patch-merge；展开 getTake；L2 摘要 = script_diff.script_diff_summary |

### 3.6 暂不接（保持本地 / 后续 ticket）

scene/shot/take 下拉切换与新建、mark 状态循环、Delete last、Next take、speaker override 持久化、memo 保存、导入/导出、observer 数、电平表——均无对应端点，留本地 state，不在 1.J–1.L 范围。

## 5. 跨平台

后端 `pathlib` + env 配置，无平台分支。前端纯 Web。入口 `python -m backend.api` 在 Windows/macOS 一致。

> **dev 摩擦提示（codex P2）**：`ADMIN_TOKEN` 缺失时随机生成 + 每次重启变化，配合前端 localStorage 存 token 会导致重启后需重填。runbook 要求开发期设固定 `ADMIN_TOKEN` env 规避。
