## §3 输入契约

### 数据类型选择

⚠ 本 spec 决策：用 `@dataclass(frozen=True)` 而非裸 `dict`，caller 传结构化对象。任务描述里「dict 输入返回 dict 输出」的原意是「I/O 由 caller 负责」（不让 pipeline 直接读写 DAL），不是强制用裸 dict。frozen dataclass 提供类型提示和字段存在性保证，测试 fixture 更清晰，且 caller（1.H）组装时有补全提示，收益大于与裸 dict 的统一性。Lead 若坚持裸 dict，改用 TypedDict 即可，接口语义不变。

### 签名

```python
@dataclass(frozen=True)
class L2Input:
    take_id: int
    scene_id: int
    take_number: int
    transcript_segments: list[dict]
    script_lines: list[dict]          # 当前场次剧本行，可为空列表
    previous_notes: list[str]         # 历史 take 的 script_diff_summary，可为空列表
```

### 字段说明

**`take_id`**（`int`，必填）：用于 logging 和错误消息的上下文标识，pipeline 不写库，但异常和日志打这个 ID 能让 caller（1.H）追溯来源。

**`scene_id`**（`int`，必填）：同 take_id，仅作上下文标识。

**`take_number`**（`int`，必填）：同上。

**`transcript_segments`**（`list[dict]`，必填，可为空列表）：ch1 的全量已 diarized 片段，每个元素结构：

```python
{
    "speaker": str | None,   # 说话人标签，如 "SPEAKER_00"；未知时为 None
    "text": str,             # 转录文本
    "start_frame": int,      # 16kHz 基准帧偏移
    "end_frame": int,        # 16kHz 基准帧偏移
}
```

由 1.H 从 `dal.list_segments(take_id, ch=1)` 组装而来，字段顺序按 `start_frame` 升序。空列表合法（take 内无语音片段时），pipeline 返回空 `line_matches`。

⚠ 本 spec 决策：`transcript_segments` 总字符数上限 **2500 字符**（`sum(len(s["text"]) for s in segments)`）。理由：`n_ctx=4096 tokens`（llm-backend-selection v0.3，llm-service-design v1.1），留出 system prompt（约 300 token）+ script_lines（约 500 token）+ output（512 token），transcript 安全预算约 2700 token，中文约 1 char ≈ 1 token，保守取 2500 字符作为 MVP 上限；精确 token 计数待 1.H 实测后复审（见 §10 Q1）。超限时截断策略：**保留最后 N 字符**（取末尾段落，因为 take 末尾更可能包含关键台词），截断后在 user message 头部标注 `[WARNING: transcript truncated to 2500 chars, earliest segments dropped]`。

**`script_lines`**（`list[dict]`，必填，可为空列表）：当前场次剧本行（最新版本），每个元素结构：

```python
{
    "line_no": int,             # 台词行号，来自 script_lines.line_no
    "character": str | None,   # 角色名；舞台指示行为 None
    "text": str,                # 台词文本
}
```

⚠ 本 spec 决策：`script_lines` 传入**已过滤的相关行**（由 1.H 决定范围，比如当前场次全部台词行），pipeline 不做二次过滤。`script_lines` 总字符数上限 **1000 字符**，超限时 1.H 在组装输入前截断（pipeline 不处理 script_lines 截断，只处理 transcript 截断）。空列表时 L2 仍运行，但 `line_matches` 会全部为空（无剧本可比较），`script_diff_summary` 标注 `null`。

**`previous_notes`**（`list[str]`，必填，可为空列表）：历史 take 的 `script_diff_summary` 字符串，供模型感知同场次早期 take 的偏差趋势。空列表时 prompt 省略该节。上限 3 条（只取最近 3 个 take），每条最多 200 字符，超限截断（1.H 组装时处理）。

## §4 输出契约（L2Output）

⚠ 本 spec 决策：返回值同样用 `@dataclass(frozen=True)` 而非裸 dict。caller（1.H）用 `dataclasses.asdict(output)` 序列化为 JSON 写库，成本为零，类型安全收益大。

### 反推依据

按 `sqlite-schema v0.3.2` 反推：

- `takes.script_diff`（TEXT，L2 输出，JSON 文本）
- `take_line_matches`（整张表由 L2 Pipeline 写入，见 schema §2.9）

`status`、`notes`、`performer_issues`、`audio_quality` 字段均来源于 NP Pipeline 或手动 mark（schema §2.2 字段说明），**不属于 L2Output**。`onset-llm-ux §Take 记录数据结构` 同样将 `script_diff` 单独标 LLM，其他字段标 Ch2/手动，与此一致。

### dataclass 定义

```python
@dataclass(frozen=True)
class LineMatch:
    line_no: int           # 对应 script_lines.line_no
    diff_type: str         # "match" | "missing" | "substitution" | "insertion"
    detail: str | None     # 偏差明细文本（改词/加词时填实际转录片段）；match 时为 None

@dataclass(frozen=True)
class L2Output:
    script_diff_summary: str | None    # 人类可读偏差摘要，→ takes.script_diff（JSON 顶层字段）
    line_matches: list[LineMatch]      # 逐行比对结果，→ take_line_matches 表（由 1.H caller 写）
```

### 字段说明

**`script_diff_summary`**（`str | None`）：整个 take 的偏差摘要，自然语言，一两句话。`script_lines` 为空时为 `None`（无剧本可比较）。1.H 在写 `takes.script_diff` 时将整个 `L2Output` 序列化为 JSON：

```json
{
    "script_diff_summary": "台词基本匹配，第3行二号演员漏说「请别」两字。",
    "line_matches": [
        {"line_no": 1, "diff_type": "match", "detail": null},
        {"line_no": 3, "diff_type": "missing", "detail": "漏词：「请别」"}
    ]
}
```

**`line_matches`**（`list[LineMatch]`）：逐行比对结果，可为空列表。`diff_type` 枚举取值与 `take_line_matches.diff_type` CHECK 约束完全一致：

| 值 | 含义 |
|---|---|
| `match` | 台词完全匹配 |
| `missing` | 演员漏说该行部分或全部台词 |
| `substitution` | 演员说法与剧本不符（改词） |
| `insertion` | 演员说了剧本没有的台词 |

⚠ 本 spec 决策：`insertion` 类型的 `line_no` 填 `-1`（无对应剧本行），`detail` 填实际转录文本。理由：`take_line_matches.line_id` 不允许 NULL（外键），1.H 在写库时遇到 `line_no=-1` 的 insertion 跳过写 `take_line_matches`（只保留在 `takes.script_diff` JSON 里），这比放宽 schema 约束成本低（见 §10 Q3）。

## §5 Prompt 设计

### System prompt

从 `TASK_CONFIG["l2_take"]["system"]` 读取模板，追加格式约束。最终 system message：

```
{TASK_CONFIG["l2_take"]["system"]}

输出格式要求（严格遵守）：
- 只输出合法 JSON，不要 markdown 代码块，不要注释，不要额外解释。
- JSON schema：
  {
    "script_diff_summary": "<str 或 null>",
    "line_matches": [
      {"line_no": <int>, "diff_type": "<match|missing|substitution|insertion>", "detail": "<str 或 null>"}
    ]
  }
- line_matches 只列出 script_lines 提供的行，不自创行号。
- insertion 类型（演员台词剧本无对应行）line_no 固定填 -1。
```

⚠ 本 spec 决策：MVP 不加 few-shot 示例。理由：system prompt 约束格式已足够明确，few-shot 增加 token 消耗（512 token output budget 按 TASK_CONFIG 已锁），调优需要真实数据；先跑 baseline 取得失败样本，再在 1.G 后续 prompt-tuning ticket 里加（见 §10 Q2）。

### User message 模板

按以下结构组装（Python f-string 风格）：

```
## 剧本台词（场次 {scene_id}）

{script_lines_block}

## Take {take_number} 转录记录

{transcript_block}

{previous_notes_section}

请按要求输出 JSON 偏差报告。
```

**`script_lines_block`** 格式：

```
[行{line_no}] {character}：{text}
```

`character` 为 None 时输出 `（舞台指示）`。空列表时整节替换为 `（无剧本，跳过偏差检测）`。

**`transcript_block`** 格式（已截断的 segments，按 start_frame 升序）：

```
[{speaker_label}] {text}
```

`speaker` 为 None 时输出 `[未知说话人]`。每段一行。

**`previous_notes_section`** 格式（仅 `previous_notes` 非空时出现）：

```
## 历史偏差参考（最近 {N} 条）

{previous_notes[0]}
{previous_notes[1]}
...
```

### Messages 组装

```python
messages = [
    {"role": "system", "content": system_prompt},
    {"role": "user", "content": user_message},
]
```

传入 `llm_service.infer(messages, task_type="l2_take", priority=2, timeout=timeout)`。

⚠ 本 spec 决策：`timeout` 作为 `run_l2_take` 参数，默认值 `60.0`（不用 LLMService 默认的 30s）。理由：L2 是 take 结束后批处理，不在用户等待路径（1.H 异步触发，不阻塞 take.end 响应）；给出更宽的超时容忍。参数化的目的是让测试可以用小值（如 0.1s）驱动超时场景，不需要真实等待（llm-service v0.1 §3 异常清单：asyncio.TimeoutError 由 caller 决定重试）。

## §6 输出解析与错误处理

### 解析流程

```
LLM 返回 raw_text (str)
    ↓
strip_markdown_fence(raw_text)   # 剥除 ```json ... ``` 包裹
    ↓
json.loads(cleaned_text)
    ↓
validate_fields(parsed_dict)     # 字段存在性 + 枚举值合法性
    ↓
L2Output(...)
```

**`strip_markdown_fence`** 实现：若文本以 ` ```json` 或 ` ``` ` 开头，剥除首尾 fence 行。只剥一层，不递归。

### 错误处理决策

**JSON 解析失败**（`json.JSONDecodeError`）：抛 `L2ParseError("LLM response is not valid JSON", cause=exc)`。caller（1.H）决定是否重试或降级（1.G 不实现重试，见 §9）。

**字段值为 `null`**（`script_diff_summary: null` 或 `detail: null`）：合法，对应 Python `None`，不抛错。`script_lines` 为空时 LLM 返回 `null` 是预期行为。

**字段 key 缺失**（`line_matches` key 或 `script_diff_summary` key 完全不存在于 JSON dict）：⚠ 本 spec 决策：**抛 `L2ParseError`**，不用默认值兜底。理由：key 缺失通常意味着模型没有遵循输出格式约定（而非「没有 match」），用空列表兜底会掩盖格式问题；与 null 的区分让 caller 能区分「模型正确返回空」和「模型格式错误」两种情况。

**`diff_type` 枚举非法**（不在 `match/missing/substitution/insertion` 内）：⚠ 本 spec 决策：**抛 `L2ParseError`**，不归类 `unknown`。理由：枚举值直接对应 schema 的 CHECK 约束，写库时传非法值会触发 IntegrityError；在 pipeline 层提前拦截，错误语义更清晰，不需要 caller 再做二次校验。

**`line_no` 不是整数**：抛 `L2ParseError`，同上原则。

**LLM 返回文本为空字符串**：抛 `L2ParseError("LLM returned empty response")`。

### L2ParseError 定义

```python
class L2ParseError(Exception):
    def __init__(self, message: str, cause: Exception | None = None) -> None:
        super().__init__(message)
        self.cause = cause
```

`cause` 串联原始异常（`json.JSONDecodeError` / `KeyError` 等），调用方 `except L2ParseError as e: e.cause` 可取原始异常细节。

Import 路径：`from backend.pipelines.l2_take import L2ParseError, L2Input, L2Output, LineMatch, run_l2_take`。`backend/pipelines/__init__.py` 按需 re-export，不强制要求全部 re-export。

## §7 测试设计

### DI vs 单例决策

⚠ 本 spec 决策：`run_l2_take` 签名接受 `llm_service: LLMService` 参数（依赖注入），不在 pipeline 内部调 `get_service()`。

理由：
1. 测试时直接传 `StubClient` 注入的 `LLMService` 实例，无需 mock 全局单例，与 llm-service v0.1 的测试风格一致。
2. 1.H 的 Orchestrator take.end handler 持有 `LLMService` 实例（通过 `get_service()` 取得），传进来是自然的调用链，不增加复杂度。
3. Pipeline 不持有全局状态，单元测试可并行运行不互相干扰。

`run_l2_take` 是 async 函数（因为它 `await llm_service.infer(...)`）。测试用 `pytest-asyncio`（conftest.py 已有 asyncio 相关配置时直接复用）或 `asyncio.run()`。⚠ 本 spec 决策：如果测试文件里其他测试已用 `pytest-asyncio`，复用；否则全文件用 `@pytest.mark.asyncio`，避免混用两种风格。

## §8 验收

以下条款对应 development-plan §C4 与 onset-llm-ux L2 用例反推的可测内容：

1. `run_l2_take` 接受合法 `L2Input`，返回 `L2Output`，`L2Output.line_matches` 的 `diff_type` 全部在 `{match, missing, substitution, insertion}` 枚举内。
2. LLM 返回 markdown fence 包裹的 JSON 时，pipeline 能解析成功，不抛错。
3. LLM 返回非 JSON 时，`L2ParseError` 抛出且 `e.cause` 不为 `None`（可追溯原始异常）。
4. `transcript_segments` 为空列表时，pipeline 不抛错，`L2Output.line_matches` 为空列表。
5. `run_l2_take` 不访问 DAL、不访问任何全局状态，只依赖传入的 `L2Input` 和 `llm_service` 参数（验证：可在无数据库环境下运行单元测试）。
6. `run_l2_take` 调用 `llm_service.infer` 时 `task_type="l2_take"` 且 `priority=2`（与 development-plan §1.G 子任务约定一致）。

## §10 待澄清问题

**Q1（上游冲突 · 必须 Lead 拍板）**：development-plan v0.2 §1.G 子任务文写「验证输出 status / script_diff / notes 字段」，与 `sqlite-schema v0.3.2` 字段说明（`status` / `notes` 来源是 NP / 手动 mark，不是 L2）及 `onset-llm-ux` L2 用例描述冲突。本 spec §4 按 schema/UX 反推，判定 dev-plan 子任务文是陈旧描述，**L2 不输出 status / notes**。若 Lead 有不同判断，需更新本 spec §4 并同步 sqlite-schema §2.2 字段来源标注。

**Q2（调优时机）**：MVP baseline prompt 能否达到可用的 diff 准确率？建议 1.G 完成后用 2-3 个真实 take 做 smoke test，记录失败样本。若准确率低，few-shot + prompt 调优作为独立 ticket。本 spec 不预设 few-shot 结构，调优时再决定。

**Q3（insertion 写库）**：`insertion` 类型 `line_no=-1` 时，1.H 跳过写 `take_line_matches`（因为外键 `line_id` 不允许 NULL）。这意味着演员加词的信息只存在 `takes.script_diff` JSON 里，不可按 `take_line_matches.diff_type` 检索。如果后续 QP Pipeline 需要「查所有有加词的 take」，这条信息就不可索引。Lead 需决定：是否在 schema 允许 NULL line_id 或引入 `insertion` 的特殊处理行（`line_id=NULL + diff_type="insertion"`）。本 spec 先用 `-1 + 跳过写库` 方案推进，不等 schema 变更。

**Q4（transcript 截断精度）**：本 spec §3 取 2500 字符作为 transcript 上限，是基于「中文约 1 char ≈ 1 token」的粗估。实际 Gemma 4 tokenizer 对中英文混合文本的 char-to-token 比率未实测。1.H 联通 LLMService 时应实测一个典型 5 分钟 take 的 transcript 长度 + context 实际使用量，若溢出再收紧或改用 token 计数（`llm_service` 层可扩展 `count_tokens` 接口）。

**Q5（previous_notes 来源）**：`previous_notes` 定义为「历史 take 的 `script_diff_summary`」，但 `L2Output.script_diff_summary` 在 1.H 写库时是 JSON 的顶层字段，而 `takes.script_diff` 存的是整个 JSON。1.H 在组装 `previous_notes` 时需要从 `takes.script_diff` JSON 里提取 `script_diff_summary` 字段，不是整个 JSON 字符串。这个提取逻辑在 1.H spec 里明确，本 spec 不定义。
