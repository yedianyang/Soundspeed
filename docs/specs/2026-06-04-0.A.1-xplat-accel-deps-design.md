# 0.A.1 跨平台硬件加速依赖方案（Apple Silicon + Windows CUDA）

- 状态：Draft（待评审）
- 日期：2026-06-04
- 范围：依赖管理与环境配置基建，不改 backend 功能代码
- 关联：接 0.A.0（`scripts/detect_device.py` 硬件探测）；衔接已合并的 1.x PR #11（实时 ASR + VAD + diarization + speaker）
- 验证依据：本 spec 的技术结论由 6 个并行验证 agent 实证（uv lock 实跑、PyPI/wheel index HTTP 实测、读项目实际代码），confidence 全 high。文末附证据要点。

---

## 1. 背景与问题

当前依赖管理有几处债，跨平台（mac + Windows）同步开发会放大它们：

1. **`pyproject.toml` 的 `[project.dependencies]` 是空的**。真清单在 `backend/requirements.txt`，靠 `pip install -r` + 手动 venv。`uv run`/`uv sync` 读 pyproject，跟 requirements.txt 脱节（曾导致 mac 上 `uv sync` 漏装 `python-multipart`，误判成 PR blocker）。
2. **没有 `.python-version`**。uv 抓系统最新（实测抓到 3.14），而 pyannote/torch 全栈在 3.14 上 wheel 未必齐全，需钉 3.12。
3. **torch 只有注释**（`requirements.txt` 末尾手写 `pip install torch==2.11.0+cu128 ...`），不可重现。
4. **四个加速包的 GPU 后端无自动化**：llama-cpp-python（L2 Gemma）、pywhispercpp（ASR）、onnxruntime（silero VAD）、torch（pyannote 底座）各有各的平台分流方式，纯声明式表达不全。

两端要"同步开发、同步运行"，最怕一套 uv、一套 requirements+手动 venv 漂移。所以本方案统一 uv，并把"按硬件装对加速包"自动化。

## 2. 目标与非目标

**目标**
- 两端从一份入库声明（pyproject + uv.lock + `.python-version` + `install_accel.py`）可重现地装出各自的加速环境。
- 统一 uv 工作流，消除 pyproject/requirements 双轨。
- 借 `detect_device.py` 探测结果自动选 CUDA/Metal/CPU 后端装对的包。

**非目标**
- 不支持 Intel Mac / Linux（只锁 Apple Silicon arm64 + Windows x86_64 两个目标）。
- **不改经纬现有能跑的 Windows 环境**。他 3060Ti + 现方案已 463 passed，迁 uv 是可选优化，由他择机，不强制打断。
- 不在本方案里换 diarization 引擎（pyannote 4.0 保留；onnx 统一后端仅作附录候选）。

## 3. 平台与硬件矩阵

| | 境熙 | 经纬 |
|---|---|---|
| 平台 | macOS Apple Silicon（arm64） | Windows x86_64 |
| GPU | Apple Metal（统一内存） | **NVIDIA RTX 3060Ti** |
| 架构 | — | **Ampere，sm_86，compute capability 8.6** |
| diarization 加速 | CPU（pyannote MPS 死路，见 §5） | CUDA |

> 注：早期验证 agent 因看不到 `device-detected.json`（per-machine、gitignored），从 torch 的 `+cu128` pin 推测经纬是 Blackwell（sm_120/RTX 50 系），并据此设计了"官方无 cu128 wheel → 第三方/源码编译"的复杂分支。**实际经纬是 3060Ti（Ampere）**，该分支不适用——见 §5 llama-cpp。Blackwell 特例留作 future（当前两台机器都不是）。

## 5. 逐包说明与验证发现

### 5.1 torch / torchaudio — uv 平台分流（high，已实证）

`[project.dependencies]` 里写 **plain `==2.11.0`，绝不写 `+cu128`**（mac PyPI 没有 local 版，写 `+cu128` 会让 mac 解析直接失败）。`+cu128` 这个 local 标签由 index 路由产生。PEP 440 下 plain `==2.11.0` 能匹配带 local 的 `2.11.0+cu128`。

实证：临时目录跑 `uv lock`（uv 0.11.16），单份 `uv.lock` 同时含 mac arm64 的 `torch 2.11.0`（pypi.org）和 Windows 的 `torch 2.11.0+cu128`（download.pytorch.org/whl/cu128）两条带平台 marker 的条目——**一份 lock 就是 universal lock**，两端 `uv sync` 各取所需。

`--torch-backend=auto` 只在 `uv pip` 接口可用，对 `uv lock`/`sync` 无效，本项目走 lock/sync，用不上。

### 5.2 pyannote.audio + torchcodec — 新增硬依赖（high）

- **engine.py 无需改**：`backend/diarization/engine.py` 只在 `torch.cuda.is_available()` 为真时 `.to("cuda")`，否则落 CPU 分支，从不 `.to("mps")`；pyannote 4.0 自身只自动升 CUDA、绝不自动用 MPS。mac 上天然走 CPU，设计正确。MPS 死路根因是 `aten::_fft_r2c` 未实现且会产出错误分离结果。
- **新坑：pyannote 4.0.4 硬依赖 `torchcodec>=0.7.0` + 系统级 ffmpeg**，当前 `requirements.txt` 没列。必须补 `torchcodec==0.14.0`（torch↔torchcodec 是运行期 ABI 耦合，metadata 不声明，两端必须一起钉；torch 2.11.0 配 torchcodec 0.14.0）。
- **ffmpeg 是系统级依赖**（不随 pip wheel 来）：mac 本机已有 8.0.1；Windows 端要单独装 ffmpeg 并入 PATH，写进 onboarding，否则 pyannote import 期就崩。
- mac torch wheel ~80MB（CPU/MPS）；Windows cu128 wheel 打包 CUDA runtime 数 GB。

### 5.3 silero-vad / onnxruntime — 推翻原设计（high）

原设计：Windows 用 `onnxruntime-gpu`（CUDA EP）、mac 用 CoreML EP。**验证推翻，改为两端都用普通 `onnxruntime`**：

- `backend/vad/detector.py` 走 `load_silero_vad(onnx=True)`，silero 6.2.1 在该函数里**硬编码 `force_onnx_cpu=True`**，`OnnxWrapper` 不接受 providers 参数。所以这条路径在 mac 和 win 上**都只跑 CPU EP，EP 分流对它完全无效**。
- 这是上游有意设计：silero 是 ~1-2M 参数小模型，512 样本/帧、线程钉死 1/1 的流式调用，逐帧 GPU 派发是延迟瓶颈，CPU 反而快。
- 结论：silero VAD 两端留 CPU，`detector.py` 不改。给 VAD 上 GPU 得绕开 wrapper 自写 InferenceSession + 复刻状态处理，收益存疑，挂起等 benchmark。
- **别用 `silero-vad[onnx-gpu]` extra**：它声明的 `onnxruntime-gpu` 无平台 marker，mac 上会试图装（无 macOS wheel）而失败。onnxruntime 当直接依赖、用普通包。
- onnxruntime macOS wheel 是 `macosx_14_0_arm64`，要求 **macOS 14.0+**（uv 解析需 `MACOSX_DEPLOYMENT_TARGET=14.0` 或机器 macOS≥14）。

### 5.4 llama-cpp-python — L2 Gemma（high）

- **mac arm64**：PyPI sdist 源码编译默认带 Metal（`GGML_METAL` 在 APPLE 默认 ON），无需任何 CMAKE_ARGS。实证本机 0.3.23 编出 `libggml-metal.dylib`，`llama_supports_gpu_offload()` 返回 True。前提：Xcode Command Line Tools + cmake。
- **Windows 3060Ti（Ampere sm_86）**：abetlen 官方 **cu124 wheel 原生支持 Ampere**（cu124 含 sm_86 SASS），直接可用。cu124 wheel 自带 CUDA 12.4 runtime，配经纬的 12.8 驱动向后兼容正常跑。**不需要 cu128/第三方/源码编译**——那是针对 Blackwell 的，经纬不是。
- **经纬现状保留**：他现在能跑就别动。本方案推荐路径（Ampere → 官方 cu124 wheel）是他择机迁 uv 时的目标态，不强制立即重装。
- uv 坑：默认 `index-strategy=first-index`，给 `--extra-index-url` 时会先命中 PyPI sdist 编出 CPU 版且不报错。所以 `install_accel.py` 必须用**确切 wheel 直链**装，再 `assert llama_supports_gpu_offload()` 验证真带 GPU。

### 5.5 pywhispercpp — ASR（high）

- **mac arm64**：默认 `pip/uv pip install pywhispercpp==1.5.0` 的预编译 wheel **已带 Metal**（实测打包 `libggml-metal.dylib`，`use_gpu` 默认 true），无需额外步骤。CoreML 不值得（v1）：要放弃预编译 wheel 改源码编译 + 每模型转 `.mlmodelc`，只换 encoder ~3x 且叠在已有 Metal 上。
- **Windows CUDA**：官方预编译 win wheel 是**纯 CPU**（无 cuda/cublas）。要 CUDA 必须源码编译，**关键坑：`GGML_CUDA=1 pip install` 会静默抓 CPU wheel 忽略 flag，必须加 `--no-binary pywhispercpp`** 才真编译。前提：CUDA Toolkit 12.x（nvcc）+ VS2022 C++ 工具链。
- **经纬现状保留**：他若现在 ASR 能跑（CUDA 编译版或 CPU 版），不动。迁 uv 时 `install_accel.py` 走源码编译分支。

```toml
[tool.uv]
environments = [
  "sys_platform == 'darwin' and platform_machine == 'arm64'",
  "sys_platform == 'win32'",
]

[[tool.uv.index]]
name = "pytorch-cu128"
url = "https://download.pytorch.org/whl/cu128"

[tool.uv.sources]
torch = [
  { index = "pytorch-cu128", marker = "sys_platform == 'win32'" },

  { index = "pytorch-cu128", marker = "sys_platform == 'win32'" },
]
# mac（darwin）不匹配 marker → 回落默认 PyPI，拿 2.11.0 CPU/MPS wheel
```

## 7. `scripts/install_accel.py` 设计

`uv sync` 之后跑 `python scripts/install_accel.py`，读 `detect_device` 结果，按平台/硬件兜底编译型包。职责单一：只处理声明式表达不了的 GPU 变体。

- **读** `device-detected.json`（无则先跑 `detect_device.collect()`）。
- **macOS arm64**：基本 no-op（llama-cpp / pywhispercpp 的 PyPI wheel 已带 Metal）。可选 `assert llama_supports_gpu_offload() == True`。
- **Windows + NVIDIA 检测到**：
  - llama-cpp-python：按 `compute_cap` 分支。`< 12.0`（含 3060Ti 的 8.6）→ 官方 cu124 wheel 直链：`uv pip install --reinstall --no-deps llama-cpp-python --index-url https://abetlen.github.io/llama-cpp-python/whl/cu124`。`>= 12.0`（Blackwell，future）→ 源码编译 `-DGGML_CUDA=on` 或第三方 wheel。装后 `assert llama_supports_gpu_offload()`。
  - pywhispercpp：源码编译 `set CMAKE_ARGS=-DGGML_CUDA=on & set GGML_CUDA=1 & set FORCE_CMAKE=1` + `uv pip install --reinstall --no-deps --no-build-isolation --no-cache-dir pywhispercpp==1.5.0`。失败要硬报错，不静默回落 CPU。
- **关键纪律**：重装 GPU 变体一律 `--reinstall --no-deps`（否则 uv 重解析依赖树，把刚装的又换回 PyPI 默认 wheel）；用确切 wheel 直链（绕开 first-index 静默装 CPU sdist 的坑）；装完 assert GPU offload，把静默退化变成硬失败。

## 8. `detect_device.py` 要补的探测项

现有探测 OS/Python/NVIDIA/CUDA toolkit(nvcc)/Metal/llama-cpp offload。补三项编译前提门控：

- **Xcode CLT（mac）**：`xcode-select -p` 返回路径且退出码 0 即装好。
- **VS Build Tools（Windows）**：不能用 `shutil.which('cl')`（dev shell 外找不到），用 `%ProgramFiles(x86)%\Microsoft Visual Studio\Installer\vswhere.exe -products * -requires Microsoft.VisualStudio.Component.VC.Tools.x86.x64 -property installationPath`。
- **cuDNN（Windows）**：onnxruntime-gpu 已放弃，此项可缓；若 future 用到 CUDA EP 再补。
- nvcc 已有 `detect_cuda_toolkit` 覆盖，无需重复。

## 9. `.python-version` = 3.12 入库

仓根加一行 `3.12` 并入库，根治 uv 抓系统 3.14。注意：torch/torchaudio/torchcodec 都有 cp314 wheel，若 uv 抓到 3.14 仍能解析（反而更隐蔽），所以必须显式 pin。可选配 `requires-python = ">=3.12,<3.13"` 上限加固。

## 10. `requirements.txt` 处置

改成 uv 派生产物（实落地命令）：`uv export --no-hashes --no-emit-project --no-dev --no-annotate --frozen -o backend/requirements.txt`，头部自带「autogenerated by uv」注释，勿手改。保留给 CI / 不用 uv 的过渡期兜底。两套手维护的清单必然漂移，不允许。

**实现回写（2026-06-05）**：原稿说「`uv export` 是 per-platform（mac 导出不带 +cu128）」，实测不对。因 `[tool.uv] environments` 锁了 darwin-arm64 + win32 两个目标，export 出来是 universal 清单，带平台 marker，Windows 行就是 `torch==2.11.0+cu128 ; sys_platform == 'win32'`，mac 行是 `torch==2.11.0 ; ...darwin`，两端通用。但清单不含 index 指令，Windows 直接 `pip install -r` 仍需先从 cu128 index 装好 torch/torchaudio（PyPI 无 +cu128）。跨平台真相源仍只活在 `uv.lock`，两端 `uv sync` 读同一份入库 lock 才是首选。

## 11. 分阶段落地

- **阶段 1（境熙 / mac，本方案落地）**：写 pyproject deps/sources/index、加 `.python-version=3.12`、删旧 .venv `uv venv --python 3.12`、`uv sync`、写 `scripts/install_accel.py`（mac 分支基本 no-op）、补 `detect_device` 探测项、`uv lock` 生成入库 lock、验证 `pytest` 463 passed 基线 + `detect_device` 摘要。
- **阶段 2（经纬 / Windows，可选、非紧急）**：经纬现有环境能跑，迁 uv 是优化不是急务。他择机切：删旧 venv → 确认 `.python-version` → `uv sync` → `install_accel.py`（Windows NVIDIA 分支：llama-cpp cu124 wheel、pywhispercpp 源码编译）→ `pytest` 463 passed。**切 uv 当天两端要一起做**（避免一端 lock 一端 pip 漂移）；在此之前不强制打断他。
- **阶段 3（评估，不一定做）**：onnx 统一后端候选，见附录 A。

## 13. 关键风险与未决

| 项 | 状态 | 说明 |
|---|---|---|
| 经纬 GPU 架构 | **已定** | 3060Ti / Ampere sm_86 → llama-cpp 官方 cu124 wheel 即可 |
| onnxruntime-gpu 取舍 | 建议放弃 | silero 写死 CPU，-gpu 无收益且有同装冲突坑；两端普通 onnxruntime |
| torchcodec + ffmpeg | 必须补 | requirements 漏了 pyannote 4.0 的硬依赖；Windows 要单独装 ffmpeg 入 PATH |
| torchcodec Win ABI | **待经纬实测** | cu128 index 上没有 torchcodec，只能从 PyPI 取 win_amd64 wheel（已核：PyPI 有 0.14.0 cp310–cp314 win wheel）。它与 cu128 torch 是运行期 ABI 耦合，需经纬 `uv sync` 后 `import torchcodec` + 实跑一段 pyannote decode 坐实。崩则降版 / 换 decode 后端，**加 `[tool.uv.sources]` 路由到 cu128 此路不通**（index 无此包，lock 会 404） |
| 切 uv 时机 | 待协调 | 两端同步切，谁先跑首次 `uv lock` 入库、`.python-version` 落地 |
| macOS 14+ | 约束 | onnxruntime wheel 要 macOS 14.0+（mac 部署目标） |
| torch 2.11.0 钉死 | 待定 | cu128 index 天花板就是 2.11.0；要 bump 须两端同步 + torchcodec 同步 |
| mac CLT-only 编译 | 待 spike | 干净 CLT-only（无完整 Xcode）机能否编 llama-cpp Metal，未验 |
| pywhispercpp Win 编译 | 路径脆 | 1.5.0 源码编译可能要改 setup.py；团队可考虑自建内部 wheel 缓存 |

## 附录 A：onnx 统一后端候选（未来评估，不在本方案）

diarization 当前用 pyannote.audio 4.0（torch 栈）：Windows CUDA 好，mac 只能 CPU（MPS 死路），跨平台体验割裂。一个未来候选是迁到 **sherpa-onnx**（onnx 格式 segmentation pyannote-3.0 + WeSpeaker embedding，onnxruntime 后端）——onnxruntime 覆盖 CPU/CUDA/CoreML/NNAPI，**Windows 走 CUDA EP、mac 走 CoreML EP，一套后端跨平台统一**。代价是推翻刚落地的 pyannote 4.0 选型，须单独 spike 评估，不在本方案范围。

## 附录 B：验证证据要点

- torch universal lock：临时目录实跑 `uv lock`（uv 0.11.16），一份 lock 含 mac `2.11.0`@pypi 与 win `2.11.0+cu128`@cu128 两条带 marker 条目。
- llama-cpp wheel index：HTTP 实测 abetlen 仅 cu121/cu124/metal 存活（cu125-130 全 404）；cu124 含 Ampere SASS。
- pywhispercpp mac Metal：wheel 内实测 `libggml-metal.dylib`（816KB）。
- silero force_onnx_cpu：读 silero-vad 6.2.1 源码 `load_silero_vad` 硬编码。
- pyannote engine.py：读 `backend/diarization/engine.py` 确认 cuda-else-cpu，无 mps。
- torchcodec/ffmpeg：pyannote 4.0.4 依赖树 + PyPI wheel 核实。
